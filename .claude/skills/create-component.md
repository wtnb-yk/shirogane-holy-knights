# コンポーネント作成スキル

コンポーネントを新規作成・大幅改修する際に使用する。`.claude/rules/architecture.md` のコンポーネント分割規約に最初から準拠した設計で作る。

## 作成前の設計フェーズ（必須）

コードを書く前に、以下を整理してユーザーに提示する:

1. **責務の洗い出し**: 作成する機能に含まれる責務を列挙する
2. **コンポーネント分割案**: 1責務=1コンポーネントの原則で分割し、ファイル一覧と各ファイルの責務を表にまとめる
3. **行数見積もり**: 各コンポーネントが150行以下、JSXが50行以下に収まるか事前に判断する
4. **Server/Client境界**: どのコンポーネントが `"use client"` かを明示する
5. **データの流れ**: props のバケツリレーが深くならないか確認する

## 分割の判断基準

以下に該当する場合は分割する:
- 異なる更新頻度を持つ部分がある（静的ヘッダー vs 動的カウンター）
- 独立して再利用できる塊がある（カード、リストアイテム、フォーム部品）
- Server/Client境界をまたぐ（Client部分を最小のコンポーネントに切り出す）
- 150行を超えそう、またはJSXが50行を超えそう

以下の場合は分割しない:
- 単一の責務で一連の流れなら行数が多くても無理に分割しない
- 1箇所でしか使わない小さな表示ブロックを過度に切り出さない

## コーディングルール

- ファイル名: kebab-case（`stream-card.tsx`）
- コンポーネント名: PascalCase（`StreamCard`）
- export: named export。default export は page.tsx / layout.tsx のみ
- パスエイリアス: `@/` → `src/`
- デフォルト Server Component。ブラウザAPI使用時のみ `"use client"`
- プロトタイプが存在する場合はその構成に従う。ただしデータはデータ層から取得し、ハードコードしない

## 実装フェーズ

設計案をユーザーが承認した後、以下の順で実装する:

1. 末端の子コンポーネントから作成（依存が少ない順）
2. 各ファイル作成後、行数が規約内か確認してから次に進む
3. 親コンポーネント（オーケストレーター）を最後に作成
4. page.tsx（薄いSSG層）を作成
